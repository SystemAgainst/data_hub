# ТЗ: Информационная система для ИП (MVP)

> Облегчённая версия. Один разработчик + AI-ассистент. ~5 пользователей на старте.
> Цель — запустить рабочую версию за 1–2 недели, потом улучшать по обратной связи.
>
> Связанные документы:
> - [DESIGN.md](./DESIGN.md) — UI-kit, цвета, шрифты, компоненты
> - [ROADMAP.md](./ROADMAP.md) — план развития по версиям
> - [designBook.html](./designBook.html) — дизайн-бук

---

## 1. Что делаем и зачем

Веб-приложение — единая точка входа для партнёров по бизнесу. Сейчас информация раскидана по Google Drive, Яндекс Диску и WhatsApp. Партнёры (люди старшего возраста, низкая цифровая грамотность) теряются. Нужен простой хаб, где каждый видит свои проекты, документы и (в будущем) финансы.

**MVP должен уметь:**
- Логин по email + пароль
- Три роли: админ, участник, гость
- Список проектов и карточка проекта
- Заглушки разделов «Сколько денег» и «Где деньги»
- Открываться по ссылке на телефоне и компьютере, работать быстро

**Чего MVP НЕ делает (сознательно):**
- Не работает офлайн
- Не имеет мобильного приложения (только сайт, который можно «добавить на главный экран»)
- Не хранит файлы сам — только ссылки на Google Drive / Яндекс Диск
- Не имеет чатов, комментариев, истории изменений
- Не реализует реальные финансы (только UI-заглушки)
- Не парсит сторонние сайты (это отдельный модуль, не для MVP)

---

## 2. Пользователи

**Админ** — владелец бизнеса. Создаёт проекты, добавляет участников, назначает роли, видит всё включая финансы.

**Участник** — партнёр по конкретному проекту. Видит только проекты, к которым его привязали. Видит финансы своих проектов. Ничего не редактирует.

**Гость** — человек со ссылкой-приглашением или демо-доступом. Видит интерфейс, но без чувствительных данных. Нужен для показа системы новым людям.

---

## 3. Сценарии (пользовательские истории)

1. Админ заходит, видит все проекты, создаёт новый, добавляет участника по email.
2. Участник получает письмо со ссылкой, регистрируется, видит свои проекты на главной.
3. Участник открывает проект, видит описание, статус, ответственных, ссылки на документы.
4. Админ заходит в раздел «Сколько денег» — видит заглушку «Раздел в разработке».
5. Гость заходит по демо-ссылке, видит интерфейс с замоканными данными.

---

## 4. Страницы и что на них

**`/login`** — форма входа: email, пароль, кнопка «Войти».

**`/`** (главная, требует логина) — приветствие по имени, список карточек проектов пользователя, кнопка «Создать проект» (только админ).

**`/projects/[id]`** — карточка проекта: название, описание, статус, ответственные, список ссылок на документы, блок финансов (видимость по роли).

**`/finances/total`** — заглушка «Сколько денег». Заголовок, иконка, текст «Раздел появится в следующей версии».

**`/finances/where`** — заглушка «Где деньги». Аналогично.

**`/admin/users`** (только админ) — список пользователей, кнопка «Пригласить», смена роли, удаление.

**`/admin/projects/new`** и **`/admin/projects/[id]/edit`** — формы создания и редактирования проекта.

## 5. Модель данных (5 таблиц)

```
users
  id, email, password_hash, name, role, created_at
  role: 'admin' | 'member' | 'guest'

projects
  id, name, description, status, created_at, updated_at
  status: 'active' | 'paused' | 'archived'

project_members
  project_id, user_id, role_in_project
  (связь many-to-many: какие пользователи к каким проектам привязаны)

project_links
  id, project_id, title, url, created_at
  (ссылки на документы в Google Drive / Яндекс Диск)

finances (заготовка, пустая пока)
  id, project_id, type, amount, currency, description, date
  type: 'income' | 'expense' | 'balance'
```

Этого достаточно для MVP. Когда понадобятся комментарии, история, файлы — добавятся новые таблицы, существующие не пострадают.

## 6. Стек

**Один фреймворк на всё: Next.js (App Router) + TypeScript**

Почему: React-страницы и API-эндпоинты в одном проекте, нет отдельного бэкенда, отличная поддержка AI-ассистентами (огромный объём документации и примеров), деплой на Vercel в один клик.

**Стили: Tailwind CSS**

Почему: классы прямо в разметке, не нужно держать отдельные CSS-файлы, AI-ассистенты пишут на нём идеально.

**База данных: Supabase (PostgreSQL + Auth)**

Почему: бесплатный тариф покрывает твой масштаб на годы, авторизация (email+пароль, хеширование, сессии) уже встроена — не пишешь руками, RBAC через row-level security — пишешь правила один раз, забываешь, дашборд с таблицами в браузере — удобно смотреть данные.

**Деплой: Vercel**

Почему: `git push` на GitHub → автодеплой за минуту, бесплатный тариф более чем достаточен, HTTPS и домен из коробки.

**Что НЕ используем:**
- Никакого Docker
- Никакого отдельного backend (Express, NestJS, Fastify)
- Никакого Redis, никаких очередей
- Никакого PWA с service workers (пока)
- Никаких state-менеджеров (Redux, Zustand) — встроенного React state хватит
- Никаких UI-библиотек типа MUI или Ant Design — слишком много готового стиля, который потом тяжело менять

**Итого зависимостей:** Next.js, React, TypeScript, Tailwind, Supabase client, lucide-react (иконки). Всё.

## 8. Безопасность (короткий чек-лист)

Не нужен индустриальный security-аудит. Нужны базовые меры:

- Пароли хеширует Supabase Auth — руками ничего не пишем
- Все защищённые страницы проверяют сессию через middleware (один файл)
- Проверка ролей делается через Row Level Security в Supabase — пишется на уровне БД, обойти нельзя
- Все секреты в `.env.local`, не коммитятся в git (в `.gitignore`)
- На Vercel секреты задаются через интерфейс, не лежат в коде
- HTTPS включается автоматически на Vercel

Чего сознательно НЕТ в MVP: rate limiting, CSRF-токены отдельно (Next.js даёт защиту по умолчанию), audit logs, 2FA. Это придёт, когда понадобится.

## 9. UX-принципы для пожилых пользователей

- Все важные действия — крупными явными кнопками с подписями
- Все формы — короткие, по одному полю на смысл
- Тексты — простыми словами, без жаргона («войти» вместо «авторизоваться»)
- Каждое действие — с подтверждением для разрушительных операций («Удалить проект?»)
- Ошибки — простым языком: «Не удалось войти. Проверьте пароль» вместо «401 Unauthorized»
- Никаких всплывающих окон, тостов, исчезающих сообщений — всё видно постоянно, пока не закроют

Визуальные принципы (цвета, размеры, компоненты) описаны в [DESIGN.md](./DESIGN.md).

## 10. Деплой и хостинг

1. Создаёшь репозиторий на GitHub
2. Регистрируешься на Vercel через GitHub
3. Импортируешь репозиторий — Vercel сам видит, что это Next.js
4. Добавляешь переменные окружения (ключи Supabase)
5. Каждый `git push` в `main` = автоматический деплой

Supabase:
1. Создаёшь проект на supabase.com
2. Создаёшь таблицы через SQL-редактор в браузере
3. Включаешь Row Level Security и пишешь правила
4. Копируешь URL и ключи в `.env.local`

Всё. Никакого терминала, никакого Docker, никакого VPS.

---

## 11. Качество кода

Реалистичный минимум для одного разработчика:

- Используем TypeScript — он сам ловит половину ошибок
- ESLint и Prettier по дефолту в Next.js — не настраиваем
- Названия переменных и функций на английском, осмысленные
- Файлы компонентов — один компонент на файл
- Комментарии только там, где код неочевиден (а не пересказ кода словами)
- Никакого Clean Architecture, SOLID, DTO-mapping — это для больших команд

## 12. Риски и как их закрыть

**Риск: Запутаешься в RBAC и кто что видит.**
Решение: всё через Row Level Security в Supabase. Один раз настроил правила в БД — забыл. Невозможно случайно показать чужие данные.

**Риск: AI-ассистент пишет неконсистентный UI.**
Решение: `DESIGN.md` скидывается в начало каждой сессии с AI.

**Риск: Сломается продакшн.**
Решение: Vercel делает превью-деплой для каждой ветки. Делаешь изменения в отдельной ветке, проверяешь на превью-URL, потом мержишь в main.

**Риск: Пользователи не разберутся.**
Решение: показать систему вживую двум-трём партнёрам ещё до релиза, посмотреть где они тыкают не туда, поправить.

**Риск: Снова усложнить и забросить (как с предыдущей версией).**
Решение: жёсткое правило — если фича не нужна прямо сейчас, её нет в коде. Никаких "на будущее". План развития по версиям — в [ROADMAP.md](./ROADMAP.md).

---

## Итог

Это ТЗ описывает рабочий MVP, который реально собрать одному человеку с AI-ассистентом за 1-2 недели вечерами. Архитектура простая, но не закрывает путь к расширению: когда понадобится больше — добавишь без переделок.

Главное правило: **если фича не нужна прямо сейчас — её нет в коде.** Любая добавленная "на будущее" штука — это код, который ты будешь поддерживать вместо того, чтобы делать то, что реально пригодится.
