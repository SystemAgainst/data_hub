# ROADMAP.md — План развития

> Поэтапный план развития проекта. Каждая версия — законченная, рабочая, на продакшене.
>
> **Главное правило:** не начинать следующую версию, пока текущая не работает у реальных пользователей и не получена обратная связь.
>
> Связанные документы:
> - [TZ.md](./TZ.md) — техническое задание MVP
> - [DESIGN.md](./DESIGN.md) — UI-kit

---

## Принципы планирования

1. **Каждая версия деплоится в продакшен** — никаких "сделаю v0.2, а потом сразу v0.3, а потом выкачу".
2. **Между версиями — сбор обратной связи** — минимум неделя реального использования.
3. **План — не контракт.** Если после v0.1 пользователи говорят, что им нужны комментарии важнее финансов — план меняется.
4. **Версии не пересекаются.** Доделал одну — закрыл, начал следующую.

---

## v0.1 — MVP (1-2 недели)

**Цель:** запустить рабочую версию, чтобы партнёры могли заходить, видеть свои проекты и документы.

### Что входит

- Регистрация и логин через email + пароль (через Supabase Auth)
- Роли: админ, участник, гость
- Главная страница со списком проектов пользователя
- Карточка проекта: название, описание, статус, ответственные, ссылки на документы
- Админка: добавление/редактирование проектов, управление пользователями
- Заглушки разделов «Сколько денег» и «Где деньги»
- Адаптивная вёрстка (мобилка + десктоп)
- Деплой на Vercel + база на Supabase

### Критерии готовности

- Минимум 2 партнёра завели аккаунт и пользуются
- Получена первая обратная связь
- Нет критических багов (всё, что обнаружено за неделю использования — исправлено)

### Чего НЕТ в v0.1

- Файлы в облаке (только ссылки)
- Реальные финансы
- Комментарии
- Уведомления
- PWA
- История изменений

## v0.2 — Реальные финансы (2-3 недели после v0.1)

**Цель:** заменить заглушки финансов на работающий учёт.

### Что входит

- Раздел «Сколько денег»: общий баланс по проектам, разбивка по проектам
- Раздел «Где деньги»: распределение средств (на счетах, в кассе, в работе и т.д.)
- Внутри проекта: добавление доходов и расходов (только админ)
- Простые графики: динамика по месяцам, разбивка по категориям
- Категории операций (создаются админом)
- Валюты (минимум одна, опционально несколько)
- Экспорт операций в CSV (для отчётности)

### Сущности в БД (расширение существующей `finances`)

```
finance_categories
  id, name, type (income/expense), color, created_at

finance_operations  (то, что было таблицей finances)
  id, project_id, category_id, type, amount, currency, description, date, created_by, created_at

finance_accounts  (опционально, если нужен учёт по счетам)
  id, name, type (cash/bank/other), balance, currency
```

### Что важно учесть

- Финансы видит только админ и участники соответствующего проекта
- Все суммы — в копейках/центах (хранение целыми числами, избегаем ошибок округления)
- Дата операции — отдельно от даты создания записи
- При удалении проекта операции не удаляются (помечаются `archived`)

## v0.3 — Файлы в облаке (1-2 недели)

**Цель:** убрать зависимость от внешних Google Drive / Яндекс Диска, хранить файлы прямо в системе.

### Что входит

- Загрузка файлов в карточку проекта (через Supabase Storage)
- Список файлов с превью для изображений
- Скачивание файлов
- Удаление файлов (с подтверждением)
- Лимит размера файла (например, 10 МБ)
- Лимит общего объёма на проект (например, 500 МБ — с запасом для бесплатного тарифа Supabase)

### Что важно учесть

- Доступ к файлу проверяется на уровне Supabase Storage policies (как с RLS)
- Старые ссылки на Google Drive продолжают работать (новое — дополнение, не замена)
- При удалении проекта файлы тоже удаляются (с предупреждением админу)

## v0.4 — Коммуникация (2-3 недели)

**Цель:** дать партнёрам возможность общаться внутри системы, не уходя в WhatsApp.

### Что входит

- Комментарии в карточке проекта
- Уведомления в интерфейсе (бейдж с количеством непрочитанного)
- Email-уведомления о новых комментариях (через Supabase Auth → Resend или другой)
- История изменений проекта (кто что когда поменял)
- @упоминания пользователей в комментариях

### Сущности

```
comments
  id, project_id, author_id, content, created_at, edited_at

notifications
  id, user_id, type, payload, read_at, created_at

audit_log
  id, project_id, user_id, action, entity, entity_id, changes (jsonb), created_at
```

### Что важно учесть

- Комментарии не редактируются после 5 минут (или редактируются, но с пометкой «изменено»)
- Удаление комментария — soft delete (помечается как удалённый, текст сохраняется)
- Уведомления группируются (не «5 новых комментариев», а один пункт «5 комментариев в проекте X»)

## v0.5 — PWA и оффлайн (1 неделя)

**Цель:** превратить сайт в полноценное мобильное приложение, которое можно установить.

### Что входит

- Манифест PWA (иконка, название, splash screen)
- Service worker для кэширования (только просмотр оффлайн, не редактирование)
- Кнопка «Установить приложение» для тех, кто открывает в браузере
- Push-уведомления (опционально, требует ещё работы)

### Что важно учесть

- Не пытаться сделать оффлайн-редактирование — это огромная сложность с конфликтами синхронизации
- Кэшируем только: интерфейс, список проектов, текст карточек
- Файлы и финансы — только онлайн

## v1.0 — Публичный релиз (по готовности)

**Цель:** перевести систему из «внутреннего инструмента» в «продукт».

### Что входит

- Кастомный домен (например, `hub.your-business.com`)
- Страница входа с фирменным оформлением
- Регистрация только по приглашениям (закрытая система)
- Расширенная админка: статистика использования, активность пользователей
- Документация для пользователей (короткое видео или текстовая инструкция)
- Резервное копирование БД (Supabase делает само, но проверить и настроить)

### Критерии релиза

- Минимум 5 активных партнёров
- Все базовые сценарии отлажены
- Нет открытых критических багов
- Есть план поддержки (кто реагирует на проблемы)

## После v1.0 — возможные направления

Перечислено в порядке вероятной полезности, без сроков. Решение что делать дальше — после v1.0, на основе реального использования.

- **Парсеры недвижимости** — возврат функционала из старого проекта `data_hub`, теперь как отдельный модуль. Парсеры пишут данные в БД, в системе появляется раздел «Объекты».
- **Календарь и задачи** — связанные с проектами события, дедлайны, напоминания.
- **Мобильное приложение (нативное)** — через React Native или Expo, если PWA окажется недостаточно.
- **Многоязычность** — если появятся партнёры, которым нужен не русский.
- **Светлая тема** — если запросят.
- **Интеграции** — Telegram-бот для уведомлений, синхронизация с банком, и т.д.
- **Двухфакторная аутентификация** — когда финансовая ценность системы вырастет.
- **API для внешних интеграций** — если кто-то захочет встраивать данные в свои системы.

## Что НЕ войдёт никогда (сознательное ограничение)

Эти вещи могут показаться полезными, но добавят сложности больше, чем дадут пользы для нашего масштаба:

- ❌ Микросервисная архитектура
- ❌ Свой backend на Node.js/Python/Go отдельно от Next.js
- ❌ Kubernetes, Docker Swarm, любой оркестратор
- ❌ Собственная инфраструктура (VPS, bare metal)
- ❌ Очереди сообщений (Kafka, RabbitMQ)
- ❌ Сложная аналитика (Mixpanel, Amplitude) — встроенной Vercel Analytics достаточно
- ❌ A/B тестирование
- ❌ Машинное обучение и AI-фичи (если внезапно понадобится — через OpenAI API, не свои модели)

Эти вещи существуют для других масштабов. У нас — 5-50 пользователей, простой бизнес-инструмент. Не стартап с миллионом MAU.
